1) UUID (универсальный уникальный идентификатор) - 128-битное число,
которое используется для уникальной идентификации объектов
в различных системах. Гарантирует пренебрежимо малую вероятность коллизий
при использовании корректных алгоритмов генерации.

2) Архитектура неплохая, но можно лучше :)
2.1. Понятие "Transaction" зарезервируем для money-flow операций, создадим соответствующий
абстрактный класс, от которого унаследуем классы для перевода со счета на счет,
для пополнения счета и снятия денег через банкомат (в каждом случае при масштабировании
приложения будет проявляться своя особая логика). При этом хранить информацию обо всех money-flow
операциях будем в одной таблице базы данных. Операции заморозки/разморозки вынесем отдельно
как сервисные операции поверх счетов, которые выполняются прежде всего администраторами.
2.2. Добавим поле статуса транзакции для регистрации ошибок.
2.3. Добавим специальные классы политики контроля выполнения команд в зависимости
от уровня полномочий актора в системе.

3) Примеры
  S - классы команд, политики, акторов и т.д.
  O - интерфейсы (абстрактные классы) Command, Product, Actor, etc.
  L - абстрактный класс Actor и его наследники
  I - иерархия Event -> Transaction -> DepositTransaction
  D - построение иерархии команд

4) Использованные паттерны:
-Command (для всех основных операций)
-Strategy (внедрение логики контроля прав на исполнение команд)
-Factory (для всех типов акторов, транзакций и лог-классов)

5) См. класс AccountService (работает с бд).
Для сохранения целостности бд при конкурентном доступе будем
использовать оптимистическую блокировку (ситуации конкуренции
относительно редки). Воспользуемся инструментами Spring (аннотации
@Transactional и @Version, исключение OptimisticLockException).
Механизмы Spring будут отслеживать изменение сущности через поле version
и бросать OptimisticLockException при возникновении конфликта.
При этом для методов введем определенное количество попыток на
бесконфликтный доступ к бд.
